Descentes batch, mini batch et stochastique¶

Mise en situation¶

Je travaille pour un fournisseur d'accs internet. Je voudrais afficher une estimation du dbit thorique moyen par rappot la distance au central aux personnes qui souhaitent souscrire chez nous.

Je dispose du dbit moyen sur 6 heures pour chacun de nos abonns : des milliards de donnes !!!

Le modle n'est probablement pas compliqu mais chaque itration de la descente de gradient me prend des heures... c'est pas prs de converger ! (J'ai bien entendu normalis mes donnes, pour converger "plus vite", mais "plus vite" = "moins d'tapes" : chaque tape prenant pas mal de temps $\rightarrow$ a peut tre long quand mme...)

Descentes mini batch et stochastique¶

C'est quoi ?¶

Jusque-l, nous n'avons utilis que des descentes "batch" : on calcule l'erreur sur l'ensemble du jeu de donnes, et de l on en tire le gradient qui nous indique la descente suivre.

Cette mthode nous permet d'tre certain d'une chose (sous rserve d'avoir correctement choisi $\alpha$) : la fonction de cot dcroit chaque tape.

Changeons de mthode un instant. Au lieu de calculer l'erreur de l'ensemble du jeu de donnes, on ne calcule l'erreur que d'un seul exemple, et on applique la descente, puis on recommence pour l'itration suivante avec un autre exemple, etc...

Avec cette approche, on n'est certain que d'une chose : on a appliqu la descente par rapport un seul exemple, on fait effectivement mieux, pour cet exemple uniquement. Il est possible d'avoir globalement mieux, mais si cet exemple tait un peu erronn ou biais non.

Une autre consquence est qu'on va probablement tourner autour du minimum, sans vraiment converger.

# On choisit une fonction simpliste : f(x) = x + 1
def f(x): return x[:,0] + 1.0
def cost(x, y, theta0, theta1):
    errs =  x[:,0] * theta1 + theta0 - y
    return np.sum(errs**2)/x.shape[0]

samples = 16
X = np.arange(samples).reshape(samples,1) + 1
Y = f(X).reshape(samples,1)
# on fausse quelques donnes
Y[3,0] += 15
Y[8,0] -= 20

def gradient_descent(x, y, batch_size, alpha = 0.001, epochs = 10):
    m = x.shape[0]
    Xtmp = np.concatenate([np.ones((m,1)), x], axis = 1)
    
    theta = np.zeros((Xtmp.shape[1], 1))
    costs = []
    
    for e in range(epochs):
        # A chaque itration, on ne traite qu'une partie des donnes
        for j in range(0, m, batch_size):
            costs.append(cost(Xtmp, y, theta))
            Xbatch = Xtmp[j:j+batch_size, :]
            Ybatch = y[j:j+batch_size, :]
            grad = np.dot(Xbatch.T, np.dot(Xbatch, theta) - Ybatch)/m
            theta = theta - alpha * grad
    costs.append(cost(Xtmp, y, theta))
    return theta, costs

def cost(x, y, theta):
    return np.sum((np.dot(x, theta)-y)**2)/x.shape[0]

fig = plot.figure(figsize=(10,5))
ax = fig.add_subplot(121)
theta_batch, costs_batch = gradient_descent(X, Y, samples)
ax.plot(np.arange(len(costs_batch)), costs_batch)

Comme prvu, on constate bien qu'en mode batch, on ne fait que descendre.

En mode "1 seul exemple", on descend le plus souvent mais la fonction de cot ne descend pas chaque tape.

On appelle ce mode le mode "stochastique" parce que normalement, il faudrait traiter les diffrents exemples un par un dans un ordre alatoire.

Un peu de vocabulaire¶

  • Une itration de la descente de gradient correspond l'tape o on met $\theta$ jour.
  • Traiter tous les exemples du jeu de donnes s'appelle une poque (ou epoch en anglais)

Dans le cadre du BGD (Batch gradient descent), une poque est une itration. Dans le cadre du SGD (Stochastic gradient descent), une poque comporte autant d'itrations que d'exemples dans le jeu de donnes.

La descente "Mini batch"¶

La descente stochastique est trs pratique : on descend globalement, pas aussi efficacement qu'en batch, mais en tous les cas les itrations prennent beaucoup moins de temps.

Un autre inconvnient (en plus des oscillations) est le temps de traitement d'une poque complte : il est plus rapide de traiter un seul exemple que tout le paquet, mais il est beaucoup plus long de traiter chaque exemple un par un que de traiter tout le paquet en bloc :

  • le produit matriciel est largement optimis en gnral (et avec numpy en particulier)
  • suivant les implmentations, les capacits de la machines peuvent tre largement exploites aussi (par exemple, distribution des calculs sur plusieurs CPU)

Pour essayer de combiner les avantages des uns et des autres, on va introduire une nouvelle mthode : le MBGD (mini batch gradient descent). Au lieu de traiter tous les exemples en bloc, ou de les traiter un par un, on va les traiter par "petits paquets".

Note concernant les tailles de mini-batch¶

Comme expliqu plus haut, un gros avantage du BGD est qu'il optimise les produits matriciels. Le MBGD en profite aussi.

Les algorithmes optimiss de produits matriciels sont pour la plupart bass sur du divide to reign : la matrice est divise en 4 blocs (2x2) et les produits sont appliqus aux sous-blocs, rcursivement.

Pour profiter vraiment de cet avantage en mode MBGD, il est plus intressant de choisir comme taille de batch une puissance de 2, tailles pour lesquelles le produit matriciel sera optimal.

def gradient_descent_iter(x, y, batch_size, alpha = 0.001, iterations = 10):
    m = x.shape[0]
    
    #Normalisation (note : le 1e-9 ne sert qu' viter une division par zro potentielle...)
    mus = np.mean(x)
    sigmas = np.std(x) + 1e-9
    
    Xtmp = np.concatenate([np.ones((m, 1)), (x-mus)/sigmas], axis = 1)
    
    theta = np.zeros((x.shape[1], 1))
    costs = []
    
    batch_start = 0
    for i in range(iterations):
        # A chaque itration, on ne traite qu'une partie des donnes
        Xbatch = x[batch_start:batch_start+batch_size, :]
        Ybatch = y[batch_start:batch_start+batch_size, :]
        grad = np.dot(Xbatch.T, np.dot(Xbatch, theta) - Ybatch)/m
        theta = theta - alpha * grad
        batch_start += batch_size
        if batch_start >= m: batch_start = 0
    return theta
